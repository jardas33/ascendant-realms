# v0.116 Desktop Spike Acceptance Contract

## Boundary

This contract describes what a future desktop engine spike must prove. v0.116 does not start that spike, choose an engine, add a dependency, create a wrapper, or change runtime behavior.

## Required Acceptance Evidence

| Area | Required Proof |
| --- | --- |
| Fresh checkout | Documented setup from repository files plus one-command validation. |
| AI-operability | Codex can create, modify, validate, build, export, benchmark, and package the representative Salto slice with minimal human interaction. |
| Editor optionality | Emmanuel can avoid routine editor operation. Any unavoidable editor step is listed, justified, and minimized. |
| Scene construction | Representative scene creation/configuration is scriptable or text-editable wherever practical. |
| Content import | Content is manifest-driven, stable-ID validated, and rejects or quarantines unknown IDs. |
| Save fixtures | v0.102 fixture manifest is consumed as read-only evidence; no live save or localStorage mutation. |
| Benchmarking | Tier S/M/L and Results-transition metrics are reported using the v0.116 scorecard fields. |
| Packaging | Windows package/export is scriptable or CLI-driven. |
| Debugging | Logs, text diffs, deterministic artifacts, and validation output are sufficient for Codex-led iteration. |
| Art boundary | Runtime art remains placeholder/fallback-owned unless a future explicit integration gate approves otherwise. |

## Required Commands For A Future Spike

The exact names may differ by engine, but the future spike must provide simple commands equivalent to:

- Setup dependencies from a fresh checkout.
- Validate imported content, save fixtures, and stable IDs.
- Generate or update the representative Salto scene.
- Build/export the project.
- Package a Windows build.
- Run benchmark scenarios.
- Emit a scorecard JSON using `docs/V0116_ENGINE_SPIKE_SCORECARD_TEMPLATE.json`.

## Benchmark Contract

The future spike must use `artifacts/desktop-spike-fixture/latest/` generated by `npm run export:desktop-spike-fixture`. Required metrics include:

- FPS average and 1 percent low.
- Frame-time p50, p95, p99, and max.
- Long-task or equivalent stall counts where applicable.
- Startup latency.
- Scene launch latency.
- Representative input/action latency.
- Results transition latency.
- Memory footprint.
- Build time and package size.
- Qualitative readability notes for HUD, minimap, unit silhouettes, Lume link, Results, and campaign frame.

## AI-First Pass Conditions

A candidate cannot pass acceptance if:

- It requires Emmanuel to manually assemble routine scenes in the editor.
- Asset registration depends on routine drag-and-drop editor actions.
- Builds or Windows packaging require undocumented editor steps.
- Project setup cannot be reproduced from repository files and scripts.
- Validation depends on visual inspection instead of deterministic checks for content/save/stable-ID boundaries.
- Debugging requires opaque binary state with no practical Codex-readable evidence.

## Visual Acceptance Notes

The future spike should assess a modern top-down RTS/RPG presentation inspired by the spirit of Warlords Battlecry but using original IP and identity. It should compare polished 2D versus 2.5D fixed-camera approaches, strong faction silhouettes, atmospheric terrain, modern lighting and VFX, a central persistent hero, tactical readability, and a polished game UI that does not look like a mobile game or development dashboard.

## Current v0.116 Result

v0.116 creates only the review docs, scorecard template, export/validation scripts, and engine-neutral fixture contract needed to make a later spike measurable.
