# v0.8.1 Visual Asset Manifest Schema

Date: 2026-05-10  
Scope: schema for visual-asset metadata. This is not a new art pass, graphics overhaul, runtime loader replacement, or gameplay change.

## Purpose

The current runtime asset manifest answers one narrow question: "Can the game find an image file to load or display?" The v0.8.1 visual asset manifest answers a different set of questions:

- What is this asset?
- Where does it live?
- Is it runtime, reference, docs-only, unused, or test-only?
- Is it placeholder, prototype, candidate, final, reference, or deprecated?
- What is its source and license review status?
- Does it fit the intended battlefield scale?
- How readable and style-consistent is it?
- Should it be allowed in production?
- Which screenshot QA targets should cover it later?

The schema is intentionally additive. It sits beside the existing runtime manifest and manual asset pipeline. It does not replace `public/assets/manifests/assetManifest.json` in this phase.

## Source File

Added source schema:

- `src/game/assets/VisualAssetManifestTypes.ts`

This file defines type-safe allowed values and manifest interfaces only. It does not load assets, change Phaser preload behavior, change CSS asset usage, or mutate gameplay data.

## Manifest Shape

```ts
interface VisualAssetManifest {
  version: 1;
  updatedAt: string;
  notes: string;
  assets: VisualAssetManifestEntry[];
}
```

`version` starts at `1` because this is the first visual metadata schema. It is separate from save data and does not require a save-version bump.

## Entry Shape

```ts
interface VisualAssetManifestEntry {
  id: string;
  filePath: string;
  category: VisualAssetCategory;
  displayName: string;
  currentStatus: VisualAssetStatus;
  sourceType: VisualAssetSourceType;
  licenseStatus: VisualAssetLicenseStatus;
  usage: VisualAssetUsage;
  usedBy: string[];
  visualFamily: string;
  scaleClass: VisualAssetScaleClass;
  intendedWorldHeightPx?: number;
  currentRenderHeightPx?: number;
  silhouetteReadability: VisualAssetReviewRating;
  styleConsistency: VisualAssetReviewRating;
  replacementPriority: VisualAssetReplacementPriority;
  notes: string;
  allowedInProduction: boolean;
  needsReview: boolean;
  screenshotQaTargets?: string[];
}
```

## Field Definitions

### `id`

Stable asset id. This should usually match a runtime manifest id, `ASSET_IDS` value, or a future manifest-only id.

Rules:

- Must be unique.
- Must be non-empty.
- Should be lowercase snake case when tied to existing runtime assets.

### `filePath`

Repo-relative path to the asset or reference.

Examples:

- `public/assets/final/units/militia_unit_sprite.png`
- `public/assets/manual/ui/Main Menu Background.png`
- `procedural:battle-map-renderer`

Rules:

- Runtime entries should point to real files unless the asset is explicitly procedural.
- Reference and docs entries may point to docs or manual source files.
- Future validation should forbid missing runtime files.

### `category`

Allowed values:

```text
hero-sprite
unit-sprite
building-sprite
unit-concept
building-concept
portrait
ability-icon
resource-icon
capture-site-icon
faction-emblem
ui-background
ui-frame
splash
terrain
vfx
audio
reference
```

The category list supports the current browser prototype and leaves room for future terrain, VFX, audio, and desktop-quality references without authorizing new assets in v0.8.1.

### `displayName`

Human-readable name for review docs and future asset gallery/reporting.

### `currentStatus`

Allowed values:

```text
placeholder
prototype
candidate
final
reference
deprecated
```

Status guidance:

- `placeholder`: clearly temporary or procedural stand-in.
- `prototype`: useful for current gameplay but not production-ready.
- `candidate`: could become production art after source/license/style review.
- `final`: production-ready with clear source/license and style approval.
- `reference`: not meant to ship as runtime art.
- `deprecated`: retained for history or migration only.

Default policy for v0.8.1:

- Current assets should not be marked `final` unless there is explicit source/license/style evidence.
- Most current file-backed art should be `prototype` or `candidate`.
- Manual source duplicates should usually be `reference`.

### `sourceType`

Allowed values:

```text
manual
generated
purchased
original
unknown
external-reference
```

Guidance:

- `manual`: manually placed into `public/assets/manual`.
- `generated`: generated by an approved tool and still requiring prompt/source review.
- `purchased`: licensed from an asset marketplace.
- `original`: internally produced original project asset.
- `unknown`: source has not been proven.
- `external-reference`: reference-only material from outside the project.

### `licenseStatus`

Allowed values:

```text
owned
generated-review-needed
licensed
unknown
reference-only
do-not-ship
```

Guidance:

- Runtime assets with `unknown` must set `needsReview: true`.
- Runtime assets must not use `reference-only` or `do-not-ship`.
- `final` assets cannot have unknown source or license.
- `generated-review-needed` is allowed for prototypes but should not imply production readiness.

### `usage`

Allowed values:

```text
runtime
manual-reference
docs-reference
unused
test-only
```

Guidance:

- `runtime`: loaded or displayed by current app code.
- `manual-reference`: preserved source/reference file, not currently selected by runtime.
- `docs-reference`: referenced by documents or prompt plans.
- `unused`: present but not currently used or intentionally reviewed.
- `test-only`: used only by test or screenshot tooling.

### `usedBy`

A list of code/data/doc targets that explain why the asset exists.

Examples:

- `src/game/scenes/BootScene.ts`
- `src/game/entities/Unit.ts`
- `src/game/ui/hudPanels/HeroHudPanel.ts`
- `src/game/styles/main-menu.css`
- `docs/CINDERFEN_VISUAL_REWORK_SPEC.md`

Runtime entries should not have an empty `usedBy` list.

### `visualFamily`

Freeform grouping for visual review.

Examples:

- `Free Marches hero`
- `Free Marches infantry`
- `Ashen Covenant enemy`
- `Cinderfen resource site`
- `Prototype UI kit`
- `Main menu background`

### `scaleClass`

Allowed values:

```text
hero
infantry
ranged
caster
large-enemy
enemy-commander
small-monster
building-small
building-medium
building-large
capture-site
ui
portrait
terrain
audio
reference
unknown
```

This field links assets back to the v0.8 scale/readability audit without changing runtime radii, footprints, capture-site radii, or camera behavior.

### `intendedWorldHeightPx`

Optional intended visual height in the current browser prototype camera scale.

Examples from the v0.8 scale audit:

- Hero target height: about 82.65 px.
- Common infantry: about 43.8 to 47.45 px.
- Brute: about 58.4 px.
- Enemy commander: about 65.7 px under current normal-unit scaling.

This field is metadata only. It must not drive gameplay.

### `currentRenderHeightPx`

Optional current rendered visual height when known.

This is especially useful for sprites whose runtime render size is derived from unit radius or building footprint rather than source image dimensions.

### `silhouetteReadability`

Allowed values:

```text
low
medium
high
unknown
```

This is a review field, not an automated score.

### `styleConsistency`

Allowed values:

```text
low
medium
high
unknown
```

Use this to record whether the asset matches Ascendant Realms' current/future visual language.

### `replacementPriority`

Allowed values:

```text
low
medium
high
critical
```

Initial guidance:

- Terrain/capture-site identity gaps are likely high or critical.
- Runtime unit/building inconsistency is likely medium to high.
- Pure manual references can be low if they are not shipped.

### `notes`

Short human-readable notes. Keep this concise enough to make manifest reviews practical.

### `allowedInProduction`

Boolean production gate.

Rules:

- `false` for unknown-license assets.
- `false` for reference-only or do-not-ship assets.
- `true` only when source/license status permits shipping.

For v0.8.1, many current runtime assets should remain `false` until source/license review is explicit.

### `needsReview`

Boolean review gate.

Rules:

- `true` for unknown source/license.
- `true` for generated assets awaiting review.
- `true` for assets with low style consistency or critical replacement priority.
- `false` only when no immediate metadata/style/legal review is needed.

### `screenshotQaTargets`

Optional list of screenshot target ids that should exercise this asset.

Examples:

- `main-menu-desktop`
- `tutorial-proving-grounds-desktop`
- `campaign-map-desktop`
- `cinderfen-crossing-launch-desktop`
- `cinderfen-watch-pressure-desktop`
- `results-victory-desktop`

This field is optional in the schema because Phase 6/7 will define the screenshot QA plan and harness later.

## Browser Prototype Support

The schema supports the current browser prototype by:

- Allowing current runtime sprites, icons, portraits, backgrounds, and UI-kit assets.
- Allowing procedural placeholders via path conventions such as `procedural:battle-map-renderer`.
- Recording current render-height metadata without changing runtime render code.
- Keeping visual review metadata separate from save files and gameplay content.

## Future Desktop Support

The schema also supports future desktop-quality work by:

- Keeping broad categories for terrain, VFX, audio, and references.
- Requiring source/license/status metadata before production claims.
- Recording scale classes that can survive a later 2.5D or 3D pipeline decision.
- Supporting screenshot QA linkage before large art replacement.

This does not authorize desktop packaging, engine switching, 3D rewrite, external generated assets, paid APIs, or large binary asset commits.

## Validation Targets For Later Phases

Future validation should check:

1. Asset ids are unique.
2. Runtime `filePath` entries exist unless explicitly procedural.
3. `category` is valid.
4. `currentStatus` is valid.
5. `sourceType` is valid.
6. `licenseStatus` is valid.
7. `usage` is valid.
8. Runtime assets cannot have `licenseStatus` `reference-only` or `do-not-ship`.
9. Runtime assets with `licenseStatus` `unknown` must set `needsReview: true`.
10. `final` assets cannot have unknown source or license.
11. `intendedWorldHeightPx` must be positive when present.
12. `currentRenderHeightPx` must be positive when present.
13. `replacementPriority` is valid.
14. Runtime assets should have non-empty `usedBy`.
15. Deprecated assets should not be runtime-used.

Phase 4 should integrate this validation into `npm run validate:content` if it can be done cleanly without broad pipeline churn.

## Non-Goals

This schema does not:

- Add art.
- Generate art.
- Download art.
- Add external assets.
- Change gameplay.
- Change save compatibility.
- Replace the current runtime asset manifest.
- Require pixel-perfect screenshots.
- Implement a desktop port or engine switch.
- Promote any enemy pressure behavior.
