# v0.79 First Signature System Priority

Status: docs-only priority lock.

This document records `Lume Network` as the first future major original signature gameplay pillar that should receive a dedicated deep-design gate. It does not approve runtime implementation during v0.79.

## Approved First Priority

Approved first future signature system:

```text
Lume Network
```

The Lume Network should become the first major original system design gate because it connects the game's world identity to its RTS/RPG decision-making.

## Why Lume Network Comes First

The Lume Network is approved as first priority because it:

- connects the world's lore to gameplay;
- transforms mines and strategic sites into a distinctive territorial system;
- reinforces the Jardas identity;
- gives races different relationships with the living land;
- builds on existing resource-site foundations;
- can later support Living Mines, rival pressure, Dynamic Battlefield Oaths, campaign choices, Retinue recovery, stronghold links, and faction asymmetry.

## Future Example Directions

Possible future examples:

- connect a quarry to a shrine for defensive wards;
- connect two mines for deeper extraction;
- connect a spring to Salto for recovery;
- defend a junction site to preserve bonuses;
- sever an enemy link to weaken a fortress;
- allow each race to interact differently with the same network.

These examples are not implementation instructions. They are design probes for a later dedicated gate.

## Relationship To Jardas

The Lume Network should make the Jardas role legible:

- awakening sites;
- binding sites responsibly;
- defending living land links;
- guiding power without treating the land as disposable fuel;
- choosing when short-term advantage harms long-term stewardship.

The system should make the protagonist feel oath-bound, not merely like a generic resource manager.

## Smallest-Fun-Slice Requirement

The later Lume Network gate must identify the smallest fun prototype slice before runtime work begins.

That gate should answer:

- which existing site types are involved;
- what one player decision is fun;
- what one enemy pressure case tests it;
- what UI surface explains it;
- what is saved, if anything;
- how Tutorial/no-reward routes stay simple;
- how replay and farming are protected;
- how to test it without broad campaign rewrite;
- what is explicitly deferred.

## Prohibited During v0.79

Do not:

- implement the Lume Network;
- add Lume links to runtime battles;
- change resource-site behavior;
- rename Aether/Mana runtime copy;
- add save fields;
- add maps, factions, races, units, buildings, or classes;
- add art, VFX, audio, or assets;
- start Living Mines;
- start Dynamic Battlefield Oaths;
- start Salto Stronghold runtime work;
- start desktop engine or packaging work.

## Priority Outcome

The next safe system-design target is a dedicated v0.81-style Lume Site Network Prototype Specification and Smallest-Fun-Slice Gate, after v0.80 plans copy migration safely.
