# v0.116 Architecture Decision Record

## Status

Reviewed architecture direction only. No engine has been selected, no desktop engine project has been created, no wrapper has been added, and no runtime gameplay, save, stable-ID, network, or art integration change is authorized by v0.116.

## Decision

Move from browser-visual-expansion work into an engine-neutral desktop spike preparation posture. The browser prototype remains the current playable source of truth, but v0.115 ended with a RED browser performance gate, so future desktop decisions should be based on a reviewed architecture packet and a representative Salto fixture instead of more runtime art expansion in the browser.

The next approved engine work, if explicitly started later, must prove that Codex and automation can create, modify, validate, build, export, benchmark, and package the representative slice with minimal Emmanuel editor operation.

## Non-Decision

- No Godot, Unity, Unreal, Electron, Tauri, or other engine path is selected.
- No desktop engine dependency is added.
- No engine project files, wrapper app, packaging shell, or runtime bridge are created.
- No game behavior, save schema, stable ID, content ID, art import path, or network surface changes.
- No v0.117 work starts.

## Evidence Basis

- v0.091 established reusable architecture, desktop criteria, vertical-slice scope, staged experiments, and save/content reuse plans.
- v0.101 froze portable content export and stable ID posture.
- v0.102 proved save translation as a contract, not a runtime migration.
- v0.107 defined a Salto reference-only art/composition boundary.
- v0.108 defined the representative battle benchmark posture.
- v0.109 through v0.115 proved the browser battle-loop performance issue is real and remains RED after focused rescue passes.

## AI-First Development Criterion

Future engine spikes must be scored heavily on AI-operability:

- Reproducible setup from repository files and scripts.
- Scriptable or text-editable scene creation and configuration where practical.
- Manifest-driven content import with automatic validation.
- Editor-optional routine workflow for Emmanuel.
- Builds, exports, tests, benchmarks, and Windows packaging runnable through simple scripts or CLI commands.
- Fresh-checkout setup and one-command validation.
- Asset registration without manual drag-and-drop editor work in routine development.
- Explicit minimization of any unavoidable manual editor steps.
- Debuggability through text diffs, logs, CLI tools, and deterministic artifacts.

## Visual Ambition

The target visual direction should evoke the spirit of a super-cool 2026 evolution of Warlords Battlecry while remaining an original game with its own IP, factions, lore, assets, UI, mechanics, and visual identity. The future benchmark should test a modern top-down RTS/RPG presentation, especially polished 2D versus 2.5D fixed-camera approaches, with strong faction silhouettes, atmospheric terrain, modern lighting and VFX, a central persistent hero, tactical readability, and no generic mobile-game or developer-dashboard appearance.
